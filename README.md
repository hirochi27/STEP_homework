# 課題2
**木構造を使えば O(log N)、ハッシュテーブルを使えばほぼ O(1) で検索・追加・削除を実現することができて、これだけ見ればハッシュテーブルのほうが優れているように見える。ところが現実の大規模なデータベースでは、ハッシュテーブルではなく木構造が使われることが多い。その理由を考えよ。**

* 大規模なDBにおけるハッシュテーブルの場合、保存するデータも多く、ハッシュ値が衝突する可能性が高くなるため、O(1)での検索・追加・削除ができなくなってしまう。（多段ハッシュにすれば大丈夫、、？）
  
* ハッシュテーブルは空のバケットが存在するため、無駄にメモリ空間を占領していまう。対して木構造の場合、空のメモリを占有することはないため効率よく格納できる。

* ハッシュテーブルの場合、再ハッシュが必要になるため一瞬だが計算に時間がかかってしまう。宿題１の簡単なコードでも数秒かかることがあるため、大規模なDBの場合、計り知れない時間がかかってしまう。対して、木構造の場合は1から構造を変える必要がない。

* 木構造の方が方が構造的にシンプルになるため、大規模になればなるほど扱いやすい

# 課題3
**Webページのキャッシュ管理をO(1)でできるデータ構造を考える（LRU）**

### ハッシュテーブルと双方向の連結リストを使う
* 連結リストはキャッシュ数と同じ数を保つ
* 連結リストの順番でアクセスの新旧を判断したい


**基本のながれ（保存したいページの情報をItemとする)**

1. 連結リストの頭にItem(Ｗｅｂページの情報)を追加
2. 連結リストからあふれた一番後ろのItemからハッシュテーブルにアクセスし情報を削除
3. キャッシュに追加したいItemをハッシュテーブルに追加

**同じページに複数回アクセスする可能性があるため**
* ハッシュテーブルにはWebページとURLの他に、連結リストに入れたItem自体を保存しておく

1. 連結リストに追加する前に、ハッシュテーブルに既に値が存在しないか確認する
2. 存在した場合、連結リストの順番を変えたい（昔保存されたItemを最新の位置に持ってきたい）
3. そのため、目的のバケットに保存されたItemの"next"と"prev"を連結させることで、連結リストから一度Itemを削除
4. 再び連結リストの頭にItemを繋ぎ直すことで順番を変えられる

* ハッシュテーブルとは別に、キャッシュしたい大きさと同じキューの仕組みを使用する
* しかし、キューだと同じページに複数回アクセスがあった場合に、3種類のページをキャッシュできなくなるので連結リストを使用する
- キュー　：　new A→B→C old　　の時、Bにアクセスしたら B→A→B になってしまう


- 最新のアクセス3つを保存するキャッシュを考えた場合

- 基本的の動き：キャッシュにページを保存したい場合はキューに追加し、出てきた古い値と対応するハッシュテーブル内の情報を削除。
- 保存したい値は普通にハッシュテーブルに追加
- これだけだったらO(1)で実現できる

- 同じページに複数回アクセスした場合：ハッシュテーブルに保存したい値が既にないか探す。
- 見つかった場合、連結リスト内にも値が存在するため、見つけ出し、連結リストの一番最初に移動させる（これに最悪O(N)かかりそう）
